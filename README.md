# Shoe Store

Учебный проект по автоматизации работы магазина обуви. Репозиторий содержит серверную часть на ASP.NET Core, desktop-клиент на WPF и общую библиотеку моделей, DTO и вспомогательных классов.

Проект покрывает базовые сценарии учёта магазина: авторизацию сотрудников, управление ролями и персоналом, магазинами, товарами, поступлениями, продажами, бронированиями и скидками.

## Состав репозитория

- `Backend/` - ASP.NET Core backend с HTTP API, JWT-аутентификацией и EF Core.
- `Frontend/` - WPF-клиент для работы с системой.
- `Classes/` - общая библиотека сущностей, DTO и helper-классов.
- `db_create.sql` - SQL-скрипт создания базы данных.
- `postman.json` - коллекция запросов Postman для ручного тестирования API.
- `Diagrams/` - draw.io-диаграммы по предметным областям.

## Основные возможности

- авторизация сотрудников и выдача JWT-токена;
- управление ролями и сотрудниками;
- добавление магазинов;
- создание и поиск товаров;
- хранение характеристик товара;
- учёт поступлений на склад;
- оформление продаж и возвратов;
- создание и отмена бронирований;
- создание скидок и привязка скидок к товарам.

## Структура решения

### Backend

Серверная часть публикует API в пространстве маршрутов `api/*`.

Основные контроллеры:

- `api/auth` - авторизация и обновление токена;
- `api/employees` - работа с сотрудниками;
- `api/roles` - работа с ролями;
- `api/shops` - работа с магазинами;
- `api/items` - создание, получение и поиск товаров;
- `api/incomes` - поступления товаров;
- `api/sales` - продажи и возвраты;
- `api/reservations` - бронирования;
- `api/discounts` - скидки.

По коду используются роли `Admin`, `Seller`, `Merchandiser`.

### Frontend

Клиент написан на WPF и взаимодействует с backend по HTTP. В текущем состоянии в интерфейсе явно реализован сценарий входа и открытие админской панели.

Админские окна, присутствующие в репозитории:

- `AdminPanel`
- `EmployeeEditor`
- `SalesEditor`

### Classes

Библиотека содержит:

- доменные сущности (`Item`, `Sale`, `Reservation`, `Shop`, `Employee` и др.);
- DTO для обмена между слоями;
- общие helper-классы и перечисления.

## Требования для запуска

- Windows
- .NET SDK 8.0 или новее с поддержкой target framework `net8.0-windows`
- SQL Server / SQL Server Express
- сертификаты разработки ASP.NET Core для запуска по HTTPS

## Конфигурация

### База данных

По умолчанию backend использует строку подключения из `Backend/appsettings.json`:

```json
"DefaultConnection": "Server=.\\SQLEXPRESS;Database=shoe_store;Trusted_Connection=True;TrustServerCertificate=True;Encrypt=False"
```

Если у вас другой экземпляр SQL Server, перед запуском измените строку подключения под своё окружение.

### Адрес backend для WPF-клиента

Во frontend адрес API захардкожен в `Frontend/Helpers/ServerContexts/ServerContext.cs`:

```csharp
internal static readonly string BackendHostUrl = "https://localhost:7217";
```

Если backend запускается на другом адресе или порту, это значение нужно обновить.

## Подготовка базы данных

Есть два варианта инициализации:

1. Использовать `db_create.sql`.
2. Запустить backend с корректной строкой подключения: в `ApplicationContext` вызывается `Database.EnsureCreated()`, поэтому база будет создана автоматически при первом старте приложения.

Важно: в проекте также есть папка `Backend/Data/Migrations`, но фактически в коде используется именно `EnsureCreated()`, а не применение миграций при запуске.

## Быстрый старт

1. Убедитесь, что SQL Server доступен и строка подключения в `Backend/appsettings.json` настроена корректно.
2. Проверьте ссылки на общую библиотеку.
3. Запустите backend.
4. При необходимости скорректируйте URL backend во frontend.
5. Запустите WPF-клиент.

Команды для запуска:

```powershell
dotnet run --project .\Backend\Backend.csproj
dotnet run --project .\Frontend\Frontend.csproj
```

## Авторизация

- backend использует JWT Bearer Authentication;
- после входа сервер возвращает токен и роль пользователя;
- часть методов защищена через `[Authorize]` и `[Authorize(Roles = ...)]`.

Замечание: в текущей реализации срок жизни JWT-токена очень короткий - 1 минута.

## Тестирование API

Для ручной проверки запросов можно использовать `postman.json`.

В коллекции есть примеры запросов для:

- создания магазина;
- создания товара;
- обращения к базовому API-маршруту;
